# 文档模板

每种文档的模板和写法要点。

---

## PRD（Product Requirements Document）

PRD 是项目的单一事实来源。**先写 PRD 再动手。**

### 模板

```markdown
# {项目名} — Product Requirements Document

## 概述

一句话说清这是什么、解决什么问题。
明确定位：个人工具 / 开源项目 / 商业产品。

## 核心理念

- **关键词** — 一句话解释
- **关键词** — 一句话解释
- 3-5 条，确立设计哲学

## 用户画像

一段话描述：目标用户身份、工作习惯、核心痛点。

## 核心流程

<!-- 用实际的示例展示主场景 -->
<!-- UI 截图 / CLI 输出 / 使用流程 / 目录结构 等 -->
<!-- 让读者（包括 AI）秒懂"用起来什么样" -->

## 功能需求

### P0 — 能用（MVP）

| 功能 | 状态 | 说明 |
|------|------|------|
| xxx  | ❌   | 具体描述，不模糊 |

### P1 — 好用（体验增强）

| 功能 | 状态 | 说明 |
|------|------|------|

### P2 — 智能 / 高级

| 功能 | 状态 | 说明 |
|------|------|------|

### P3+ — 扩展 / 生态

| 功能 | 状态 | 说明 |
|------|------|------|

#### 复杂功能展开（按需）

<!-- 表格一行说不清的功能在这里展开 -->
<!-- 可包含：策略、配置项、行为矩阵 -->

## 非功能需求（按需）

| 需求 | 说明 |
|------|------|
<!-- 用具体数字：启动 < 200ms、支撑 10K 条等 -->
<!-- 内容项目可省略此节 -->

## 不做的事

- 明确列出不做的功能
- 划清边界，防止 scope creep

## 技术选型（开发项目）

| 用途 | 选型 | 说明 |
|------|------|------|

详细方案见 `{project}-techstack.md`。

<!-- 内容项目可省略此节或替换为"工具选型" -->

## 相关文档

| 文档 | 内容 |
|------|------|
```

### 写法要点

- **P0-P5 分层**管理优先级，不一股脑列
- **状态用 ✅/❌**，PRD 本身就是进度追踪
- **核心流程用真实示例**（截图 / 输出 / 流程图），不写抽象描述
- **"不做的事"必须有**，最容易忽略也最有价值
- **复杂功能用行为矩阵**（场景 × 条件 → 行为）
- 做完改 ✅，砍掉的移到"不做的事"
- 中文撰写，技术术语保留原文

## techstack.md `[开发]`

### 模板

```markdown
# {项目名} — 技术选型 & 项目结构

## 技术栈

| 用途 | 选型 | 说明 |
|------|------|------|

## 项目结构

<!-- 目录树 + 行内注释 -->

## 依赖清单

<!-- 列出所有直接依赖及用途 -->

## 开发阶段

### Phase 1: xxx ✅
- [x] 已完成项
- [ ] 待做项

### Phase 2: xxx
- [ ] ...
```

### 写法要点

- 目录树要及时更新，新增文件/目录时同步
- Phase checklist 完成项打 `[x]`，新阶段直接追加
- 依赖清单注明每个依赖的用途，不只列包名

## cautions.md

### 模板

```markdown
# {项目名} — 注意事项

> 踩过的坑和关键约束，避免重复犯错。

## 1. 标题

**问题：** 描述遇到了什么

**解决方案：** 怎么解决的

**参考：** `path/to/file` 或相关链接

## 2. 标题

...
```

### 写法要点

- 每个坑独立编号，只追加不重排
- 三段式：问题 → 方案 → 参考
- 活文档，踩坑时立即记录，不要攒着
