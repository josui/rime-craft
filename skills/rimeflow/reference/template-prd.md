# PRD 模板

PRD 是项目的单一事实来源。**先写 PRD 再动手。**

## 模板

```markdown
# {项目名} — Product Requirements Document

## 概述

一句话说清这是什么、解决什么问题。
明确定位：个人工具 / 开源项目 / 商业产品。

## 核心理念

- **关键词** — 一句话解释
- 3-5 条，确立设计哲学

## 用户画像

一段话描述：目标用户身份、工作习惯、核心痛点。

## 核心流程

<!-- 用实际的示例展示主场景 -->
<!-- UI 截图 / CLI 输出 / 使用流程 / 目录结构 等 -->

## 功能需求

> 已完成阶段归档至 [archive.md](archive.md)。
> 远期需求移入 [backlog.md](backlog.md) 管理。

### P0 — MVP

| #    | 内容 | 状态 | 说明 |
| ---- | ---- | ---- | ---- |
| #001 | xxx  | ❌   | 具体描述，不模糊 |

<!-- 初期可在 PRD 中写 P0~P3 所有阶段 -->
<!-- 阶段完成后：已完成 → archive、远期 → backlog、PRD 只留当前阶段 -->

## 非功能需求（按需）

| 需求 | 说明 |
|------|------|
<!-- 用具体数字：启动 < 200ms、支撑 10K 条等 -->

## 不做的事

- 明确列出不做的功能
- 划清边界，防止 scope creep

## 技术选型（开发项目）

| 用途 | 选型 | 说明 |
|------|------|------|

## 相关文档

| 文档 | 内容 |
|------|------|
| backlog.md | Backlog（改善点、Feature Ideas） |
| archive.md | 已完成功能归档 |
| product/ | 详细仕様書 |
```

## 写法要点

- **PRD 只保留当前活跃阶段**，已完成的归 archive，远期的归 backlog
- **所有功能项带全局编号** `#001`、`#002`...跨文档唯一
- **状态用 ✅/❌**，PRD 本身就是进度追踪
- **核心流程用真实示例**，不写抽象描述
- **"不做的事"必须有**，最容易忽略也最有价值
- 做完改 ✅，阶段完成后整体移入 archive
